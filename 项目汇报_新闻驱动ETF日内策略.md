## 写给修改者和评委（请先读这一段）

本系统每天开盘前自动回答一个问题：**今天 50 万虚拟资金，买哪几只 ETF、买多少、还是空仓？**  
答案以比赛要求的 JSON 格式输出，并可从网页仪表盘查看「大模型为什么这么选」。

文中若出现内部代号，含义如下（正文第一次出现也会再解释）：

| 内部叫法 | 实际含义 | 读者可理解为 |
|----------|----------|--------------|
| **大模型决策版** | 新鲜新闻 + 经济日历 + DeepSeek 大模型 + 规则风控 | **当前正式使用** |
| **纯规则版** | 不用大模型，只靠关键词和 K 线打分 | 对照组 / API 失败时的备用 |
| **提示词优化版**（原「方案 A」） | 对大模型下达指令的文案做过一轮优化 | **我们最终采用的提示词配置** |
| **经济日仓位分级**（原「40% 硬顶」） | 高重要事件日按条数限制总仓位：1–2 条≤85%，3–5 条≤75%，6+ 条≤65% | 风控规则，不是模型「建议」 |
| **评分门槛 45 分** | 所有候选 ETF 最高分不够 45 就强制空仓 | 另一道风控 |
| **量价特征** | 每只 ETF 近期的涨跌幅、成交量、趋势分等 | 技术面参考，优先级最低 |

**比赛还没开始时**：本地可以提前跑通流程做测试，**不必提交**；正式参赛后每个交易日再跑当天结果并提交即可。

## 一、项目背景与目标

### 1.1 要解决什么问题

在 A 股交易日 **09:30 开盘前**，根据**当时已经能看到的公开信息**，为 **50 万元**虚拟资金做出当日配置：

- 从 **13 只 ETF**（10 只稳健池 + 3 只进攻池，宽基/商品/行业）里选出 **最多 3 只**；
- 按 **100 股整数倍**给出买入数量；
- 或输出 **空仓**（空数组 `[]`，比赛规则允许）。

持仓假设：**开盘买入、收盘卖出**（用当日开盘价到收盘价算收益，用于复盘，不是真实下单系统）。

### 1.2 比赛要求（与本项目的关系）

智投赛道强调：

1. **必须使用大模型**参与分析与决策（不能只是传统量化公式）。  
2. **经济日历优先于新闻情绪**——两者冲突时，以经济日历为准。  
3. 输出格式固定，便于平台评测。

本项目用 **DeepSeek 大模型**读「经济日历 + 筛选后的新闻 + ETF 技术面摘要」，输出结构化结论；再用 **固定规则**做仓位上限、评分门槛等风控，保证结果可解释、可复盘。

### 1.3 硬约束（防止「作弊」）

| 约束 | 说明 |
|------|------|
| 决策时点 | 只能用 **当日 09:30 之前** 的数据 |
| 禁止 | 当日 K 线、盘中价、未来公告 |
| 资金 | 50 万；ETF 按 100 股一手 |
| 输出 | `[{ "symbol", "symbol_name", "volume" }, ...]` |

## 二、AI 智能体核心方案设计

> 对应大赛材料常见板块：**技术选型 · 智能体工作流 · 核心 AI 逻辑 / 提示词设计**

### 2.1 技术选型（用了什么、为什么）

| 类别 | 选用 | 通俗说明 |
|------|------|----------|
| 开发语言 | Python 3.13 | 财经数据与脚本生态成熟 |
| **大模型** | **DeepSeek Chat** | 比赛要求的「AI 分析」核心；按天调用，结果可缓存 |
| 财经数据 | AkShare | 免费拉 A 股 ETF 行情、财经新闻、经济日历 |
| 历史新闻库 | 自研爬虫 + SQLite | 回测用；东方财富历史资讯可落到 2026 年 3–4 月 |
| 本地存储 | CSV（行情）+ JSON（每日预测）| 便于审计、答辩展示 |
| 展示界面 | 浏览器仪表盘（本机 8765 端口）| 给评委看决策理由，非必须 |
| 定时运行 | Windows 计划任务 + 批处理脚本 | 工作日约 09:25 自动跑 |

配置文件 `.env` 中需填写 **DeepSeek API 密钥**；未配置时系统自动退回 **纯规则版**，不中断程序。

### 2.2 智能体工作流（每天发生什么）

**非交易日（周六、日）**：只复盘上一交易日盈亏，**不生成新预测**。

**交易日**流程如下：

```
09:25 左右启动
    ↓
① 复盘：上一交易日预测赚了还是亏了（开盘价→收盘价）
    ↓
② 更新 ETF 历史行情（写入本地 CSV）
    ↓
③ 抓取盘前新闻（约 60 小时窗口内的财经资讯）
    ↓
④ 程序自动筛新闻（去掉「资金榜快讯」等噪音，保留有板块、有事件的新闻）
    ↓
⑤ 加载「经济日历」（当天及前一日重要数据/政策发布时间）
    ↓
⑥ 调用 DeepSeek：输入「日历（在前）+ 新闻摘要 + 各 ETF 涨跌量」→ 得到 JSON 结论
    ↓
⑦ 规则层合并：是否空仓、仓位上限、选哪几只 ETF、各买多少
    ↓
⑧ 写出两个文件：
      - xxx_submit.json  → 提交比赛用
      - xxx_full.json    → 含大模型原文、新闻、日历，供答辩
    ↓
⑨ 可在仪表盘查看（可选）
```

**信息优先级（比赛硬性要求）**：

```
经济日历  ＞  新闻（经大模型理解）  ＞  量价/技术面（程序算分）
```

### 2.3 核心 AI 逻辑与提示词设计（大模型具体干什么）

#### 2.3.1 大模型输入是什么

每天拼成一份「说明书式」长文本，主要包括三块：

1. **经济日历（排在最前）**  
   例如：中国 CPI、美国非农、LPR 利率等；标注重要性、已公布值/预期值/前值。  
2. **新闻摘要（排在第二）**  
   原始新闻可能上千条；程序先筛到约几十条，再压缩 **不超过 20 条** 给大模型，避免超长、避免费钱。  
3. **候选 ETF 表**  
   每只的近 1/3/5 日涨跌幅、是否放量、趋势分等（即 **量价特征**）。

#### 2.3.2 大模型必须输出什么（JSON）

大模型不直接下单，只输出「建议」，例如：

| 字段 | 含义（人话） |
|------|----------------|
| `regime` | 今天整体偏乐观 / 中性 / 偏悲观 |
| `econ_drivers` | 哪些经济事件影响了判断 |
| `news_drivers` | 哪些新闻利好/利空哪类 ETF |
| `per_etf_view` | 对每只 ETF 打 -1～+1 的分（看多/看空强度）|
| `cash_decision` | 建议空仓 / 部分买入 / 尽量满仓 |
| `position_ratio_hint` | 建议用多少比例的资金（0～100%）|
| `summary_zh` | 两三句中文总结（答辩用）|

#### 2.3.3 什么是「提示词优化版」（原方案 A）

开发过程中试过多种大模型指令写法。早期版本问题：**动不动就建议空仓**，导致几乎不交易、收益很差。

**提示词优化版**是对大模型指令文案的改进，核心三点：

1. **不要轻易空仓**——除非日历、新闻、盘面三者都说不清方向；默认「部分买入」。  
2. **打分要有力度**——真看好某 ETF 至少给 +0.5，不要清一色 0.1、0.2，否则会被程序误判为「不值得买」。  
3. **重要数据日可以谨慎，但别直接清零**——建议仓位区间约 30%～60%，后面还有规则会再压。

该版在 **2026 年 3–4 月、新闻数据完整** 的回测里收益最好，因此定为 **正式线上配置**。  
对应文件：`prompts/decider_zh.md`。

#### 2.3.4 规则层在大模型之后还做什么（评委常问）

大模型只给「建议」，程序还会强制执行以下规则（可在仪表盘看到触发记录）：

| 规则 | 人话解释 | 举例 |
|------|----------|------|
| 大模型说空仓 | 直接输出 `[]`，不买 | 很少触发（提示词优化后）|
| 建议仓位取低 | 大模型说 50%、程序算出来 98%，**取 50%** | 模型只能更保守，不能更激进 |
| **重要经济日仓位分级** | 高重要事件按条数限制总仓位（85%/75%/65%） | 即「仓位硬顶」 |
| **评分门槛 45** | 候选 ETF 综合分最高仍 &lt; 45 → 空仓 | 防止勉强买入弱信号 |
| 选券与分配 | 按综合分选前 1～3 只，按权重算股数 | 100 股整数倍 |

**举例（2026-05-25 一次实盘跑的结果）**：  
程序算大盘偏强本可用约 98% 仓位 → 大模型建议 50% → 当日经济日历触发 **分级上限（如 75%）** → 最终用 **50% 资金**（取较低者）买入 3 只 ETF。其余资金现金。

#### 2.3.5 新闻在进大模型之前怎么筛（不是 AI 瞎读全网）

程序用关键词规则先过滤，例如：

- 拒绝：「资金榜」「份额变动」类自动快讯；  
- 拒绝：看不出利好/利空哪个板块的泛泛新闻；  
- 保留：带「中标、降准、业绩预增」等具体事件、且能对应到某类 ETF 的新闻；  
- 对「连续大涨后还吹利好」的新闻降权，避免追涨。

这样大模型读到的是 **高质量摘要**，也满足比赛对「新闻情绪」的要求。

## 四、效果验证（历史回测，非实盘承诺）

### 4.1 回测怎么做的

- **时间**：2026-03-02 至 2026-04-30（43 个交易日）  
- **为何选这段**：东财财经流 + 新闻联播 + 经济日历 **数据齐全**；2026 年 5 月东财历史资讯断档，**未纳入**正式对比。  
- **资金**：50 万；每日开盘买、收盘卖；与实盘同一套决策逻辑（含两级新闻筛选 + DeepSeek 决策）。  
- **对照组**：「纯规则版」（不调 DeepSeek），用于说明大模型带来的差别。

### 4.2 主要结果（给评委看的数字）

| 模式 | 43 日累计收益 | 说明 |
|------|---------------|------|
| 纯规则版 | **+0.61%** | 基线 |
| **大模型融合** | **+1.70%** | 相对基线多约 **1.1 个百分点** |

其他曾尝试的改法（仅收紧日历定义、仅降低评分门槛）单独测试效果更差，**未采用**。

### 4.3 怎么理解这些数字

- 回测收益 **不代表**未来实盘一定盈利，仅说明「在 3–4 月、该数据条件下，当前配置优于纯规则」。  
- 胜率不是越高越好：提示词优化版有时 **主动少交易**（空仓约 10 天），换取回撤更可控。  
- 答辩时可强调：**可解释性**（每条决策有 full.json）、**符合赛规**（大模型 + 经济日历分级风控）、**工程可落地**（定时任务 + 仪表盘）。

### 4.4 和「人工读新闻」比

| 维度 | 人工 | 本系统 |
|------|------|--------|
| 耗时 | 每天约 2～3 小时读报 | 09:25 前自动跑完 |
| 覆盖面 | 几十条 | 筛后上千条里取强信号 |
| 一致性 | 随情绪变化 | 同输入同规则，可复盘 |
| 经济日历 | 容易漏 | 写入提示词最前 |
| 答辩材料 | 难留痕 | JSON + 网页可追溯 |

## 五、局限与后续计划

### 5.1 当前局限（如实写）

1. **依赖 DeepSeek 网络**：失败时退回纯规则版，质量下降。  
2. **重要经济日较多**：程序常触发分级仓位上限，可能偏保守。  
3. **新闻历史不完整**：2026 年 5 月起东财爬虫库断档，回测 5 月不可靠。  
4. **只识别周末休市**：法定节假日未单独维护，需人工勿跑。  
5. **大模型有成本**：约每日数千 token，已做缓存避免重复扣费。

### 5.2 后续可改进

- 补全节假日历；  
- 补抓 5 月新闻用于更长回测；  
- 微调「重要经济日」判定，减少不必要的分级仓位压制；  
- 比赛开始后按平台反馈迭代提示词。

## 附录 B · 经济日历在本项目中的角色（扩充）

**经济日历** = 按日列出「何时公布 CPI、PMI、利率」等宏观数据。  
与新闻相比，时间通常 **更早、更确定**，所以在提示词里 **必须写在新闻前面**。

**「经济日仓位分级」**：当系统判断当天有 **高重要** 宏观数据发布时，按事件条数限制总仓位（85%/75%/65%），即使用大模型和规则都想多买也不能突破——防止数据公布瞬间波动过大。这是 **程序硬规则**，不是大模型一句话建议。
